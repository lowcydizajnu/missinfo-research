Overview

Researchers compose studies in a browser admin panel — a feed of posts (true / false, with configurable source, headline, image, engagement metrics and comments), demographic and per-post question screens, consent / instruction / debrief text, and optional between-subjects manipulations. Participants open a public link, complete the study anonymously, and every reaction, rating, answer and (optionally) gaze sample is stored. Data exports to Excel/CSV with one row per participant.

Data model

Everything hangs off a study. Child rows link up by study_id; response tables link via session_id.

Accounts & auth

The admin panel is multi-user and invite-only — there is no public sign-up. An admin creates accounts under the 👥 Konta tab; each researcher sees and manages only studies they own (studies.owner_id), while an admin sees all.

  • Login is username + password; passwords are stored as bcrypt hashes.
  • The JWT payload is { userId, role }; every study-touching endpoint verifies ownership server-side (a researcher probing another owner's study id gets 404).
  • On first boot the server seeds one admin account from the ADMIN_PASSWORD environment variable and assigns any pre-existing studies to it. After that, ADMIN_PASSWORD is no longer consulted — change the password from inside the panel.
  • The server refuses to start without JWT_SECRET (no insecure fallback), and the login route is rate-limited.
Participants are never authenticated. Ownership scoping applies only to the admin API; the public participant routes stay open so anyone with a study link can take part.

Participant flow

A participant opening /study/<slug> progresses through (each screen is optional per study): consent → instructions → demographics → the post feed (scrolling or one-post-at-a-time "paged" mode) with reactions and per-post questions → transition screens → debrief. A session row is created at start (POST /api/session/start) and finalised at completion, capturing timings and the assigned condition.

Experimental design

Studies support a between-subjects manipulation (manipulation_json): at session start the server randomly assigns a condition, which selects a per-post content variant (e.g. headline/image A vs B) and can drive social-proof metric levels. Assignment is balanced across a queue so cells fill evenly.

Conditional logic

Beyond random assignment, a study can define deterministic within-subject branching (logic_json): rules of the form IF <demographic | condition | answer | reaction> <operator> <value> THEN <skip part | end study | go to part | hide question>. Rules are validated on save (dangling references rejected) and evaluated in the participant runtime at defined timings (after demographics, after a part, after an interaction). Skipped parts are recorded so there are no silent gaps in the data.

Translations

A study has a base language and a translations_json overlay per additional language (Polish, English, Czech, Slovak). Researchers can fill a target language automatically with the "Przetłumacz automatycznie" action, which calls the Anthropic API when ANTHROPIC_API_KEY is configured; translations can then be reviewed and edited by hand. Global UI chrome strings are overridable via locale_overrides (admin-only).

Deployment

The app is a single Node process and runs anywhere Node + a writable disk are available (a VPS, Docker, Railway, Render, …). Configure via environment variables:

VariableRequiredPurpose
JWT_SECRETyesSigning secret for admin tokens — the server won't start without it
ADMIN_PASSWORDfirst bootInitial password for the seeded admin account
ANTHROPIC_API_KEYoptionalEnables AI-assisted translation; without it the feature is disabled
DATABASE_PATHoptionalSQLite file path (default ./data/research.db)
UPLOADS_PATHoptionalLegacy disk-image fallback path (default ./uploads)
PORToptionalHTTP port (default 3000)
Secrets live only in environment variables — never commit a .env file. The database (data/) and uploads are gitignored; back them up separately.

Local development

git clone https://github.com/lowcydizajnu/missinfo-research.git
cd missinfo-research
npm install
cp .env.example .env      # then set JWT_SECRET (required) and ADMIN_PASSWORD
npm run dev               # nodemon (or: npm start)
# open http://localhost:3000/admin.html and log in as "admin"
# with the ADMIN_PASSWORD you chose — you are now the instance admin.
