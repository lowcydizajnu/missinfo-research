Metric conditions let you vary the social-proof signals that participants see under each post — the like counts, share counts, dislikes, and flags — as a between-subjects factor. Because perceived popularity and consensus are powerful drivers of belief and sharing behaviour, controlling these numbers is often as important to your study design as controlling post content. Metric conditions are defined in the **Configurator** tab and operate independently of the [A/B Manipulation](/guide/manipulation), so you can combine them for a factorial design.

## How Metric Conditions Work

When you define metric conditions, each participant session is assigned to exactly one condition. That condition specifies a **minimum and maximum range** for each engagement counter. When a participant loads a post, the platform draws a random integer from within that range and displays it as the counter value. This randomisation happens once per post per session — the number a participant sees does not change if they scroll away and return.

The resolved values are stored in the session export so you know exactly what each participant saw, not just which condition they were assigned to.

## Enabling Metric Conditions

<Steps>
  <Step title="Open the Configurator tab">
    Navigate to your study and select the **Configurator** tab.
  </Step>
  <Step title="Expand the Metric Conditions section">
    Scroll to **Metric Conditions** and click to expand it.
  </Step>
  <Step title="Toggle on Metric Conditions">
    Enable the toggle to activate the conditions editor.
  </Step>
  <Step title="Add conditions">
    Click **Add Condition** to create your first condition. Repeat for each condition you need. Most designs use two conditions (e.g. *High engagement* and *Low engagement*).
  </Step>
  <Step title="Configure each condition">
    Fill in the key, label, and min/max values for each engagement counter within each condition.
  </Step>
  <Step title="Save the Configurator">
    Click **Save**. Participant sessions created after saving will be randomly assigned to one of the defined conditions.
  </Step>
</Steps>

## Condition Fields

Each metric condition has the following properties:

<ParamField path="key" type="string" required>
  A short machine-readable identifier, e.g. `high` or `low`. This value appears in the `metric_condition` and `full_condition` export columns. Use lowercase and no spaces.
</ParamField>

<ParamField path="label" type="string" required>
  A human-readable name displayed in the builder UI and session viewer, e.g. *High engagement* or *Low social proof*.
</ParamField>

<ParamField path="likes_min" type="integer" required>
  The minimum number of likes drawn for any post under this condition (unless overridden by a post-level base value).
</ParamField>

<ParamField path="likes_max" type="integer" required>
  The maximum number of likes drawn for any post under this condition.
</ParamField>

<ParamField path="shares_min / shares_max" type="integer">
  Range for share counts. Set both to 0 to hide share counters across all posts in this condition.
</ParamField>

<ParamField path="dislikes_min / dislikes_max" type="integer">
  Range for dislike counts.
</ParamField>

<ParamField path="flags_min / flags_max" type="integer">
  Range for flag counts.
</ParamField>

## How Shown Values Are Resolved

The number displayed under a post is determined by the following priority order:

1. **Post base value** — if you have entered a non-zero value directly in the `base_likes` (or equivalent) field on the post, that exact number is shown regardless of the assigned condition.
2. **Condition range** — if no post-level base value is set, a value is drawn uniformly at random from `[min, max]` for the assigned condition.
3. **Zero (hidden)** — if both the post base value and the condition range resolve to 0, the counter is hidden entirely from participants.

<Note>
  Setting a base value of **0** on a post explicitly hides that counter for all participants on that post, overriding the condition range. To let the condition range control a counter, leave the post's base value at its default (empty or 0 depending on the field) and ensure your condition defines a non-zero range.
</Note>

## Export Columns

The following columns are added to every export row when metric conditions are enabled:

| Column | Description |
|--------|-------------|
| `metric_condition` | The key of the condition assigned to this session (e.g. `high`) |
| `likes_shown` | The actual like count displayed to this participant on this post |
| `shares_shown` | The actual share count displayed |
| `dislikes_shown` | The actual dislike count displayed |
| `flags_shown` | The actual flag count displayed |

Because the shown values are stored per-post per-session, you can model the exact number each participant saw as a continuous predictor in your analysis, rather than relying on the condition assignment alone.

## Combining with A/B Manipulation: the 2×2 Design

When both an [A/B Manipulation](/guide/manipulation) and Metric Conditions are active, every participant session is assigned to a combination of one style arm and one metric condition. The `full_condition` export column captures this combination:

| | Condition: HIGH | Condition: LOW |
|---|---|---|
| **Arm A** | `A-HIGH` | `A-LOW` |
| **Arm B** | `B-HIGH` | `B-LOW` |

Each cell in this 2×2 table is an independent between-subjects group. Assignment to both factors is random and independent — being in Arm A does not influence which metric condition a participant is assigned to.

<Tip>
  Pre-register your 2×2 design before launching, specifying which cells correspond to your treatment and control conditions. The `full_condition` column makes it straightforward to filter and compare across all four cells in your analysis.
</Tip>

## Hiding Engagement Counters Entirely

If you want to run a study where participants see **no** engagement numbers at all — no likes, no shares, no dislikes, no flags — you can disable the `show_metrics` toggle in the study **Settings** tab. This hides all counters for every participant, regardless of any metric conditions you have defined. This is useful for a no-social-proof control condition in a larger multi-study programme.

<Warning>
  Disabling `show_metrics` in Settings overrides all metric condition ranges and post base values. Participants will see no engagement counters at all. If you want some posts to show counters and others not to, set individual post base values to 0 rather than using the global toggle.
</Warning>
