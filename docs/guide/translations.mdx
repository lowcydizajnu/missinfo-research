Misinfo Research keeps three kinds of text strictly apart: the language **you** work in as a researcher, the platform's own wording **participants** see, and the study content **you wrote** (consent, posts, questions). They are stored in different places, translated by different mechanisms, and changing one never touches the others.

Getting this distinction right saves a lot of confusion later — so the first section is a map, and the rest of the page follows it.

## The three layers

| Layer | Who sees it | Where it lives | How you change it |
| --- | --- | --- | --- |
| Admin panel UI | You (the researcher) | `public/locales/admin/pl.json`, `public/locales/admin/en.json` | Language dropdown in the header or on the login screen |
| Participant interface strings | Participants | `public/locales/{pl,en,cs,sk}.json` + `locale_overrides` table | **Interface translations** modal (⚙ in the header) |
| Study content | Participants | Study columns and rows in the database, plus `translations_json` | Configurator / Settings, optionally seeded by **Translate automatically** |

<Note>
Demographic questions and their answer options are **study content**, not interface strings. You write them yourself, they live in the database, and they are translated per study — never through the interface-translations modal.
</Note>

## Layer 1 — the admin panel language

The panel ships in **Polish (default) and English**. Both live as flat key/value files in `public/locales/admin/`; Polish is the source language.

Switch it with the small language dropdown (tooltip: *Panel language*) that appears in two places:

- in the **header**, top-right, next to the ⚙ button and **Sign out**;
- on the **login card**, below the **Sign in** button — so you can read the panel in English before you have an account.

The choice is stored in your browser (`localStorage`, key `admin_lang`) and applies to that browser only. Switching reloads the page, which guarantees no already-rendered tab keeps a stale string.

<Info>
If an English key is missing, the panel falls back to the **Polish** string, never to a raw key like `header.logout`. A partially translated panel degrades to Polish rather than to gibberish.
</Info>

This setting is purely cosmetic for you. It has **no effect whatsoever** on what participants see — a panel set to English will still run a Polish study in Polish.

## Layer 2 — participant interface strings

Everything the platform says to a participant that you did not write yourself — action buttons, topic pills, navigation, generic prompts — comes from the participant locale file for the **study's** language: `public/locales/pl.json`, `en.json`, `cs.json`, or `sk.json`.

You can override any of these strings from the panel:

<Steps>
  <Step title="Open the modal">
    Click the **⚙** button in the header (tooltip: *Translations and platform settings*).
  </Step>
  <Step title="Pick a language tab">
    Tabs for 🇵🇱 Polski, 🇬🇧 English, 🇨🇿 Čeština, 🇸🇰 Slovenčina. On a non-Polish tab the Polish text is shown read-only on the left as the reference; on the Polish tab you edit Polish directly.
  </Step>
  <Step title="Edit and save">
    Type a replacement, then save. Leaving a field empty means "use the default from the locale file".
  </Step>
</Steps>

Saves do **not** rewrite the JSON files on disk. They are diffed against the file baseline and written to a `locale_overrides` table (`lang` + `key` composite key). Any value equal to the file default, or cleared, deletes its row — so the table only ever holds genuine overrides.

<Tip>
That indirection exists for a reason: on hosts with an ephemeral filesystem (Railway, containers), a file write would be silently wiped on the next redeploy. Each row is badged in the modal — **💾 DB** means "stored in the database, survives redeploy", **📄 PL.json** means "still the file default". Hovering a DB badge shows when the edit landed.
</Tip>

<Warning>
Interface strings are **global**: they apply to every study, every researcher, and every participant on the installation. Because of that, saving them requires an **admin** account (a non-admin gets a 403). Any signed-in user can open the modal and read the values.
</Warning>

<Note>
The interface-translations modal renders its own chrome (title, hint, save button, badge tooltips) in Polish regardless of your panel language. The strings you are editing are shown correctly in all four languages — only the modal's own furniture is Polish-only.
</Note>

## Layer 3 — study content

Study content is the text of the experiment itself: participant title, consent, instructions, transitions, debriefing, action labels, the Likert wording, demographic questions and their options, post questions, part labels, and the posts.

**Polish is always the source language.** Each study carries a `language` field (`pl`, `en`, `cs`, `sk`). Set it in the:

- **Configurator** tab — the compact `🇵🇱 PL / 🇬🇧 EN / 🇨🇿 CS / 🇸🇰 SK` dropdown in the study header (builder-mode studies);
- **Settings** tab — the **Study language** dropdown (older, non-builder studies).

When a study's language is anything other than Polish, translated text is not written over your Polish columns. It is stored **alongside** them, in the study's `translations_json`, keyed by language:

```json
{
  "en": {
    "consent_text": "…",
    "label_action_like": "Like",
    "demographic_questions": [{ "id": 12, "label": "Age", "options": [{ "label": "18–24" }] }],
    "posts": [{ "id": 340, "headline_a": "…", "content_a": "…" }]
  }
}
```

At runtime this overlay **shadows** the database row: participants see the translated value where one exists, and the Polish source wherever one does not. So a study set to `cs` with no translation yet is not broken — participants simply get Polish content inside a Czech interface.

<Info>
A post's `topic` stays a key (e.g. `nauka`) in the database. The topic pill is rendered from the participant locale (`topics.<key>`), which is why translating the topic key itself would break the lookup rather than fix it. Topic labels belong to layer 2.
</Info>

## Automatic translation

Setting the language to anything but Polish reveals a **✨ Translate automatically** button (it reads **Translate again** once a translation exists, next to a `✓ Translated` / `⚠ No translation` status).

Clicking it sends your study's Polish content to the Anthropic API and writes the result into `translations_json` for the target language. It also sets the study's `language` to that target.

<Steps>
  <Step title="Study-level content">
    One request covers participant title, consent, no-consent text, instructions, both transition texts, debriefing, the Likert question and its min/max labels, the four action labels (like / dislike / share / flag), the comment placeholder, plus demographic questions, post questions and part labels.
  </Step>
  <Step title="Posts">
    A second request covers the posts: headlines and content for both A and B variants, `time_ago`, and the pinned comment with its author.
  </Step>
</Steps>

Machine-readable structure is deliberately preserved: option `value` codes, question and row `id`s, Likert `scale` numbers, and `question_type` all pass through untouched — only human-visible labels are translated. Empty fields stay empty. Only **active** posts and questions are sent.

<Warning>
Automatic translation targets **English, Czech and Slovak only**, and always translates *from* Polish. There is no reverse direction: a study written in English cannot be machine-translated to Polish.
</Warning>

<Note>
If the posts request fails, the study-level translation is still saved and the posts simply keep their Polish text. You can rerun the button or fix the posts by hand.
</Note>

### The API key

Automatic translation calls the Anthropic API and needs an `ANTHROPIC_API_KEY` environment variable on the server:

```bash
# .env next to the app, or your host's environment variables
ANTHROPIC_API_KEY=sk-ant-…
```

This key is **optional**. The platform is fully functional without it — every other feature works, and you can still run multilingual studies by typing the translations in yourself. Without the key, the **Translate automatically** button returns an error asking you to set `ANTHROPIC_API_KEY` and restart the server.

<Warning>
Treat machine translation as a **first draft, not a finished instrument**. Consent forms, debriefing, and Likert anchors carry methodological weight: anchor wording changes response distributions, and an ethics committee approved a specific consent text, not an approximation of it. Have a native speaker read every translated study before it goes live.
</Warning>

## Editing translations by hand

This is the important half of the workflow, and it works exactly as you would hope: **switch the study to the target language, then edit the fields normally.**

The panel is overlay-aware in both directions. Reading a study in Czech shows you the Czech text, not the Polish source. Saving writes translatable fields back into the overlay for that language and leaves the Polish source columns intact. This holds for study-level text, demographic questions, post questions, parts, and posts alike.

<Check>
Clearing a field in a non-Polish version makes it empty **for that language** — it does not silently fall back to the Polish source. What you see in the panel is what the participant gets.
</Check>

To edit the Polish original, switch the study's language back to `pl`; you are then editing the source columns directly, and existing translations for other languages are left alone.

## Running a study in several languages

<Info>
**The workflow used in practice is one study per language.** A study carries a single `language` value, so a single study renders in exactly one language. Multi-language fieldwork is done by duplication, not by a language toggle for participants.
</Info>

<Steps>
  <Step title="Build and finalise the Polish master">
    Get the design right first — posts, conditions, demographics, consent. Every duplicate inherits it.
  </Step>
  <Step title="Duplicate the study">
    **Duplicate**, once per additional language. Settings and posts are copied; session data is not. Copies are named `… (kopia)`, `… (kopia 2)`, and so on — rename them to something like *Vaccines — CS*.
  </Step>
  <Step title="Set the language on the copy">
    Open the copy, set its language to `en` / `cs` / `sk`.
  </Step>
  <Step title="Translate">
    Click **✨ Translate automatically**, then review and correct every field by hand.
  </Step>
  <Step title="Verify, then activate">
    Preview the copy end to end before opening it to participants.
  </Step>
</Steps>

Duplication handles the awkward part for you: the copy's posts and questions get new database ids, and the ids inside `translations_json` are remapped to match, so translations that were already present keep pointing at the right rows.

Practical consequences worth planning for:

- Each language version is an **independent study** with its own slug, its own participant URL, and its own sessions. Recruit into each one separately.
- Exports are per study, so you get one dataset per language and combine them in your analysis software. Keep a stable identifier — a consistent naming scheme, or the study slug — so the language is a variable you can model.
- Changes to the Polish master after duplication do **not** propagate. Freeze the design before you duplicate; if you must change something afterwards, apply it to every copy.

<Warning>
Language is confounded with sample in this design: participants in the Czech version are Czech recruits in a Czech setting, not a random assignment to a language condition. Cross-language differences are cross-sample differences — treat them as such, and consider measurement invariance before comparing means across versions.
</Warning>
