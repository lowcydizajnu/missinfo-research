Every study in Misinfo Research has a **Settings** tab where you control how the study is identified, how it appears to participants, which integrations are active, and whether it is live. For studies built with the visual builder, the Settings tab presents the same interface as the **Configurator** — changes made in either place are identical.

<Note>
  Settings changes are saved automatically as you type. An explicit **Save** button is also available if you prefer to commit a batch of edits at once.
</Note>

## Basic Information

<ParamField path="study_name" name="Study name" type="string" required>
  The internal, admin-facing label for this study. Shown in the study list and throughout the admin panel. Participants never see this value.
</ParamField>

<ParamField path="slug" name="Slug" type="string" required>
  The URL-safe identifier appended to your instance's base URL to form the participant-facing study link (for example `/study/my-study-slug`). Must be unique across the instance. Changing the slug after a study is distributed will break existing links, so set it before sharing the URL.
</ParamField>

<ParamField path="title" name="Title shown to participants" type="string">
  The headline displayed to participants on the welcome and consent screens. If left empty, the platform uses the **Study name** value instead. Use this field when you want a participant-friendly title that differs from your internal label.
</ParamField>

<ParamField path="description" name="Description" type="string">
  Free-text notes visible only to admins and researchers. Use this field to record study purpose, version notes, or links to pre-registration documents. Participants never see it.
</ParamField>

<ParamField path="language" name="Language" type="enum">
  The content language for this study. Determines which translation overlay is applied to consent text, instructions, post content, demographic questions, and all other participant-facing strings. Available options: **PL** (Polish), **EN** (English), **CS** (Czech), **SK** (Slovak). See [Translations](/guide/translations) for how to enter or generate translated content.
</ParamField>

<ParamField path="contact_email" name="Contact email" type="string">
  An email address displayed on the **consent screen** and the **debrief screen** so participants know how to reach the research team. We recommend using a monitored institutional inbox rather than a personal address.
</ParamField>

<ParamField path="institution_name" name="Institution name" type="string">
  The name of the institution or organisation conducting the study. Displayed on the consent screen alongside the contact email.
</ParamField>

<ParamField path="custom_domain" name="Custom study domain" type="string">
  A fully qualified custom hostname (for example `study.youruniversity.edu`) on which this study is served. When set, only this study is accessible on that hostname — the regular multi-study routing does not apply. Requires your DNS and server to be configured to point the hostname to the Misinfo Research instance. Leave blank to use the default base URL.
</ParamField>

## Action Row

The row of buttons at the top of the Settings tab lets you take high-level actions on the study.

<CardGroup cols={2}>
  <Card title="Preview" icon="eye">
    Open the participant view of the study in a new browser tab without activating it. Useful for reviewing layout and content before going live.
  </Card>
  <Card title="Activate / Deactivate" icon="toggle-on">
    Toggle the study between **active** (accepting new participants) and **inactive** (shows a closed message). Redirect URLs only fire when the study is active.
  </Card>
  <Card title="Duplicate" icon="copy">
    Create a full copy of the study — including all settings, posts, and questions — under a new name. The duplicate is inactive by default and gets a new slug.
  </Card>
  <Card title="Delete" icon="trash">
    Permanently remove the study and all its data. This action cannot be undone. The platform will ask for confirmation before proceeding.
  </Card>
</CardGroup>

<Warning>
  Deleting a study removes all collected response data permanently. Export your results before deleting if you need to retain them.
</Warning>

## Display Options

<ParamField path="show_metrics" name="Show metrics" type="boolean">
  When **enabled**, like and share counters are visible to participants on each post card. When **disabled**, the counters are hidden entirely. Turn this off when you want to isolate participant responses from social-proof cues.
</ParamField>

## Integrations

### MS Clarity

Microsoft Clarity records session replays, heatmaps, and interaction metrics for your study. The integration is configured **per study** — you can enable it on selected studies without affecting others.

<Steps>
  <Step title="Enable Clarity">
    Toggle on **MS Clarity** in the Integrations section of Settings.
  </Step>
  <Step title="Paste the Project ID">
    Go to your [Clarity dashboard](https://clarity.microsoft.com), copy the **Project ID** for the project you created for this study, and paste it into the **Project ID** field.
  </Step>
  <Step title="Verify tracking">
    Activate the study and run a test session. Open the Clarity dashboard — live sessions should appear within a few minutes.
  </Step>
</Steps>

<Note>
  Each study requires its own Clarity Project ID if you want to keep session data separated. You can reuse the same Project ID across studies, but all sessions will be pooled in a single Clarity project.
</Note>

### Eye Tracking

<ParamField path="eye_tracking" name="Eye tracking" type="boolean">
  When **enabled**, the study loads **WebGazer.js** and attempts to use the participant's webcam to capture gaze data as they read posts. Participants are prompted to grant camera permission; if they decline, the study continues normally without gaze data. Enable this only when your study design explicitly requires gaze tracking, as the webcam prompt can affect consent rates.
</ParamField>

<Tip>
  Test the eye-tracking flow on the devices and browsers your participants are likely to use before activating the study. WebGazer.js performance varies across hardware configurations.
</Tip>
