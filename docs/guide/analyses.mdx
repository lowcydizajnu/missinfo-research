title: "In-app analyses"
description: "Run descriptives, t-tests, ANOVA, chi-square, correlations, regression and Cronbach's alpha on your study data without leaving the panel."
---

The **📊 Analyses** tab lets you ask a statistical question of your data while the study is still running: *does the misinformation condition change perceived credibility?*, *is my credibility scale internally consistent?*, *do older participants share more?* You pick a test, pick the variables, click **▶ Run analysis**, and get the test statistic, degrees of freedom, p-value, effect size and a one-line interpretation.

It is a quick-look instrument. It exists so that you can sanity-check a manipulation after 30 sessions instead of waiting for the full sample and a trip to R. It is not a replacement for your analysis software, and the tab says so on every result card.

Pick a study from the picker at the top first — otherwise the tab shows *Select a study from the list above.*

## Where the numbers come from

The Analyses tab does not read a separate data store. Every test runs against **the same data layer the Export tab produces** — the `Dane_surowe` (raw data) sheet, recomputed from scratch on every run. Three consequences worth internalising:

- **Results are always live.** There is no caching. Run the same analysis tomorrow and it includes the sessions that arrived overnight.
- **The variable pickers show your export column names.** The tab loads column metadata from the export preview and uses the *labelled* columns, so any header you renamed in the Export tab's column builder appears here under that new name. One source of truth for column labels.
- **The tab follows the Export tab's current options.** When you click **▶ Run analysis**, it reads the language selector and the *include preview sessions* checkbox from the Export tab and passes them along. If you have not touched those, the analysis runs on canonical data with preview sessions excluded — the same defaults as a plain export.

<Note>
Every result card shows **N (filtered)** — the number of rows in the underlying dataset — next to the test name. This is the row count *before* the test drops incomplete cases, so it will usually be larger than the N reported inside the test itself. Read both.
</Note>

## Variable types and what they unlock

Each export column carries a type, shown as a badge in the pickers:

| Badge | Type | Example columns |
| --- | --- | --- |
| 🔢 | number | `post_1_liked`, `age_code`, rating scales |
| 🏷 | categorical | `full_condition`, `age`, `post_1_reaction` |
| 📝 | text | open-ended responses |
| 🔗 | link | URL columns |

The pickers filter by type, and this is the main reason a variable you expect is missing from a dropdown. Continuous slots (means, correlations, regression, alpha) only offer **number** columns. Grouping slots only offer **categorical** or **text** columns.

<Tip>
Demographics are stored twice: `age` is the human-readable label (categorical — good for grouping) and `age_code` is the numeric code (number — good for correlations). Same for `residence`, `education` and `gender`. If a demographic will not appear in a continuous picker, you are looking at the label column; reach for the `_code` one.
</Tip>

## The tests

The sidebar lists eight tests under **Choose a test**. Selecting one clears the current parameters and result, so switch deliberately.

<AccordionGroup>

<Accordion title="📋 Descriptive statistics">
*Mean, median, SD and quartiles for a single continuous variable.*

**Fields:** Variable (one 🔢 column).

**Returns:** N (complete), Missing, Mean (M), Std. deviation (SD), Median, IQR as Q1–Q3, Min, Max, SEM.

The obvious first move for any new outcome variable — it tells you the range participants actually used, how much data is missing, and whether the distribution is anywhere near where you expected.
</Accordion>

<Accordion title="📊 t-test (Welch)">
*Compares the means of 2 groups. Welch for independent samples, standard for paired ones.*

**Fields:** *Paired samples* (checkbox), Continuous variable (🔢). Then, depending on the checkbox:
- unchecked → Grouping variable (≥2 groups) (🏷 or 📝)
- checked → Second variable (paired) (🔢)

**Independent samples** use **Welch's t-test by default** — it does not assume equal variances. This is the right default: the equal-variance (Student) t-test is only correct when group variances match, and the Welch version costs you almost nothing when they do. You get t, df (fractional, as Welch's correction produces), the p-value, Cohen's d with a magnitude label, the 95% CI of the difference, and per-group N/M/SD.

A **Levene test** (median-based) runs alongside and is reported under *Equality of variances (Levene)* with a verdict: *✓ met*, or *✗ violated (consider a non-parametric test)*. Because the test is already Welch, a violated Levene is information about your data rather than an invalidation of the result.

**Paired samples** compare two columns row by row — the same participant measured twice (e.g. a pre and post rating). You get N pairs, mean difference, SD of the difference, t, df, p, Cohen's d and the 95% CI of the difference.

<Warning>
The independent-samples test refuses to run if the grouping variable has more than two groups, and tells you to use ANOVA instead. This is a guardrail, not a bug: silently comparing the first two of four conditions would be a real methodological error.
</Warning>
</Accordion>

<Accordion title="📈 One-way ANOVA">
*Compares the means of 3+ groups. With Tukey HSD post-hoc when significant.*

**Fields:** Dependent variable (continuous) (🔢), Grouping variable (≥2 categories) (🏷 or 📝).

**Returns:** F with both df, p, η² (eta²), SS between / within, and a **Group statistics** table (label, N, M, SD per group).

**Post-hoc runs automatically** when the main effect is significant (p < .05) *and* there are at least three groups — an omnibus F only tells you *some* group differs, so the Tukey HSD table gives you the pairwise comparisons: mean difference, q, p, 95% CI, with significant rows marked. With only two groups no post-hoc appears (there is nothing to correct for — use the t-test).

Levene is reported here too, with a violated verdict suggesting *Welch ANOVA or Kruskal-Wallis* — neither of which this tab implements. That is a deliberate hand-off to your stats software.
</Accordion>

<Accordion title="🔲 Chi² test of independence">
*Whether two categorical variables are related. Cramér's V as the effect size.*

**Fields:** Row variable (🏷 or 📝), Column variable (🏷 or 📝).

The contingency table is built for you from the two columns — every observed category becomes a row or column. You get χ² with df and N, p, Cramér's V, and the full **Observed counts table**.

<Warning>
If any cell has an expected count below 5, a warning appears telling you how many cells are affected and that the result may be unreliable. Fisher's exact test is **not** available in this app — the suggested fixes are to merge categories or run the exact test elsewhere.
</Warning>
</Accordion>

<Accordion title="∼ Correlation">
*Pearson for linear continuous variables, Spearman for ranked/monotonic ones.*

**Fields:** Method (*Pearson (linear)* or *Spearman (rank)*), Variable X (🔢), Variable Y (🔢).

**Returns:** r (or r_s), N, df, p, r², and — for Pearson only — a 95% CI computed via the Fisher z transformation. Spearman gets no CI.

Pairs are kept only where **both** variables have a numeric value, and the test needs at least three complete pairs.
</Accordion>

<Accordion title="⊞ Correlation matrix">
*Pearson r between every pair of the selected continuous variables.*

**Fields:** Continuous variables (Ctrl/Cmd+click to select several) — a multi-select of 🔢 columns; at least two.

Returns a symmetric matrix of Pearson coefficients, with cells where p < .05 highlighted. Hovering a cell shows its exact p-value. The diagonal is 1.

The natural companion to a battery of rating items: run the matrix first to see what hangs together, then run Cronbach's alpha on the cluster.

<Warning>
The matrix does **no** correction for multiple comparisons. Ten variables means 45 tests, and at α = .05 you expect roughly two "significant" cells from noise alone. Treat the highlighting as a map of where to look, never as a set of findings.
</Warning>
</Accordion>

<Accordion title="↗ Linear regression Y ~ X">
*Simple OLS: fits Y = a + b·X, reports R² and the significance of the slope.*

**Fields:** Predictor (X) (🔢), Dependent variable (Y) (🔢).

**Returns:** the fitted equation, R², r, the slope (b) with its standard error, t(df) for b with a p-value, and the 95% CI for b.

This is **simple** regression — one predictor, one outcome. There is no multiple regression, no covariates, no interaction terms in this tab. Those belong in R/SPSS/JASP.
</Accordion>

<Accordion title="α Cronbach's alpha">
*Internal consistency of a multi-item scale (e.g. several Likert questions).*

**Fields:** Scale items (≥2 continuous; Ctrl/Cmd+click) — a multi-select of 🔢 columns.

**Returns:** α, Number of items, N complete cases, a Reliability verdict, and an **α if item deleted** table listing every item with the alpha the scale would have without it.

Alpha uses **listwise deletion** — only cases where every selected item has a numeric value count, so *N complete cases* can be well below your session count if participants skipped items.

The *α if item deleted* table is the point of running this before your main analysis: an item whose removal *raises* alpha is a candidate for being reverse-coded incorrectly, ambiguously worded, or simply measuring something else.
</Accordion>

</AccordionGroup>

## Saving and reloading an analysis

**💾 Save as…** prompts you for a name (the placeholder suggests something like *"Credibility by condition"*) and stores the analysis under **Saved analyses** in the sidebar.

What gets saved is the **template — the test type and the variable picks — not the numbers.** Clicking a saved analysis loads its configuration and immediately re-runs it against current data.

<Note>
This is the design decision that makes the tab useful during data collection. A saved analysis is a standing question, not a screenshot. Set up "credibility by condition" on day one, click it every morning, and you are always looking at today's sample — never at a stale result you forgot to refresh.
</Note>

Saved analyses live with the study, so they travel with it and are visible to anyone with access to that study. The 🗑 button next to a saved analysis deletes it after a confirmation prompt; it removes the template only and touches no data.

## A note on result language

The panel's interface is translated — Polish by default, with an English switch in the header. The **interpretation sentence** printed above each result table, and some validation error messages, are generated by the statistics engine in **Polish**, whatever language the panel interface is set to.

The numbers, tables, labels and headings around them are translated normally, and the interpretation line never contains anything the tables do not also show, so nothing is lost if you cannot read it.

## Before you publish

<Warning>
Every result card ends with the same line: **Result intended for quick exploration. Verify in R / SPSS / JASP before publication.** Take it literally.
</Warning>

The statistics engine's outputs are validated against R/SPSS to within four decimal places for the test fixtures in the project's test suite, so the arithmetic is not the concern. The concern is everything a quick-look tool deliberately leaves out:

- **No multiple-comparison correction** anywhere except the Tukey post-hoc.
- **No Fisher's exact test**, so small-cell contingency tables have no exact fallback here.
- **No Welch ANOVA or Kruskal-Wallis**, so a violated homogeneity assumption in ANOVA has no in-app remedy.
- **No multiple regression** — one predictor only.
- **No Shapiro-Wilk.** The engine's normality checking is a skewness/kurtosis heuristic, chosen because approximate Shapiro-Wilk implementations are unreliable at small n and a wrong p-value is worse than an honest rule of thumb.
- **No analysis plan, no pre-registration discipline, no audit trail.** Clicking through eight tests on the same data is exploration. Reporting the one that came out significant is p-hacking.

The intended workflow is: explore here, confirm there. When you are ready to write up, use the Export tab to download the data (**📥 Download Excel (.xlsx)** or the CSV, which is the `Dane_surowe` sheet only for a quick import into R/SPSS/Python) and re-run the analysis in your own software — the empty state of this tab tells you the same thing.
