description: "Fixes for the problems researchers and operators actually hit: startup failures, lockouts, missing images, study 404s, preview sessions and custom domains."

Most problems with a Misinfo Research instance come from one of three places: an environment variable that is missing or misunderstood, a study that is not active, or a session that was tagged as a preview. This page walks through the symptoms you are likely to see, explains what the server is actually doing, and gives the fix.

Throughout, the panel labels are the **English** ones (the admin panel defaults to Polish; the language switch is in the top-right of the header and on the login card). A few messages come straight from the server and are still Polish regardless of your panel language — those are called out where they matter.

## The server refuses to start

**Symptom:** the process exits immediately after launch and the log ends with:

```
FATAL: JWT_SECRET is not set. Refusing to start with a forgeable token secret.
```

**Cause:** `JWT_SECRET` signs the admin login tokens. There is deliberately no fallback value — a predictable secret would let anyone forge an admin token, so the server exits rather than start in that state.

**Fix:** set `JWT_SECRET` to a long random string and restart.

```bash
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
```

Put it in your `.env` (self-hosted) or in your host's environment variables (e.g. the Railway dashboard → Variables).

<Warning>
The server reads `.env` from the **working directory you launch it from**. Running `node /path/to/app/server.js` from your home directory will not pick up `/path/to/app/.env`. Start the app from the application directory (`npm start`), or set the variables in the environment directly.
</Warning>

With Docker Compose the failure looks different — Compose stops before the container is even built:

```
set JWT_SECRET in your .env (see .env.example)
```

Copy `.env.example` to `.env` next to `docker-compose.yml` and fill in `JWT_SECRET` and `ADMIN_PASSWORD`.

<Note>
Just above the fatal line you may also see: `[accounts] No users yet and ADMIN_PASSWORD not set — admin seed skipped.` That warning is not fatal on its own — the database layer loads before the admin routes, so both messages can appear in the same failed boot. Fix `JWT_SECRET` first, then re-read the section below.
</Note>

<Tip>
Changing `JWT_SECRET` on an existing instance is safe for your data but invalidates every token already issued. Everyone who was signed in is silently signed out on their next click. Do it deliberately, not casually.
</Tip>

## Nobody can sign in / the admin password is lost

The single most misunderstood variable on the platform is `ADMIN_PASSWORD`.

<Warning>
`ADMIN_PASSWORD` **only seeds the very first admin account**, and only when the `users` table is completely empty. After that first boot, login checks the bcrypt hash stored in the database. Editing `ADMIN_PASSWORD` later — in `.env`, in Railway, anywhere — has **no effect at all**.
</Warning>

On the boot where the seed runs, the log says:

```
[accounts] Seeded admin user "admin"; assigned 0 existing studies to it.
```

The username is literally `admin`. Any pre-existing studies without an owner are handed to that account, so nothing is orphaned.

### Work through this in order

<Steps>

<Step title="Check the username, not just the password">
The seeded account is `admin` — not your email, not your name. On the login card that is the **Username** field.
</Step>

<Step title="Check whether the account is disabled">
Login only matches **active** accounts. A disabled account produces exactly the same message as a wrong password — *"Incorrect username or password."* — by design, so nobody can probe which logins exist. An administrator can re-enable it from **Accounts** (**Status** column → **Enable**).
</Step>

<Step title="Have another administrator reset the password">
This is the normal path. In the header, click **Accounts**, find the row, click **🔑 Password**, and enter the new one at the *"New password (min. 8 characters):"* prompt. Passwords shorter than 8 characters are rejected by the server.
</Step>

<Step title="Only if there is no other administrator: reset at the database level">
The platform ships no password-reset command, and the seed will not re-run while any user row exists. If the last admin password is gone, you need shell access to the machine holding the database file.
</Step>

</Steps>

<Warning>
The last step edits the database directly. Take a copy of the `.db` file first, and stop the app while you do it.

```bash
# 1. Generate a hash with the same algorithm and cost the app uses (bcrypt, 12).
node -e "console.log(require('bcryptjs').hashSync('your-new-password', 12))"

# 2. Write it onto the admin row (adjust the path to your DATABASE_PATH).
sqlite3 ./data/research.db \
  "UPDATE users SET password_hash='<paste-the-hash>', is_active=1 WHERE username='admin';"
```

Run step 1 from the application directory so `bcryptjs` resolves. Then restart the app and sign in with the new password.
</Warning>

### Other sign-in oddities

<AccordionGroup>

<Accordion title="I was working and got thrown back to the login screen">
Login tokens expire after **24 hours**. When a token expires the panel receives a 401 and signs you out automatically. Sign in again — nothing is lost, the Configurator autosaves.
</Accordion>

<Accordion title="Suddenly no login works, even the correct one">
The login endpoint is rate-limited: **20 attempts per 15 minutes per IP address**. Once the window is exhausted, further attempts are rejected regardless of the credentials, and the login card does not tell you that this is what happened. Wait 15 minutes and try once, carefully.

If this happens constantly with only one or two people using the panel, everyone's attempts are probably being counted against the same IP — see [Rate limiting behind a proxy](#rate-limiting-behind-a-proxy).
</Accordion>

<Accordion title="I want researchers to register themselves">
They cannot. Accounts are invite-only: *"Accounts are created by an administrator (no self-registration). Each researcher sees only their own studies; the administrator sees all of them."* Create them from **Accounts** → **New account**.
</Accordion>

</AccordionGroup>

## Auto-translation reports a missing API key

**Symptom:** clicking **✨ Translate automatically** returns *"Translation error: …"* wrapping this message (it is emitted by the server in Polish even when your panel is in English):

> Brak klucza API Anthropic. Ustaw zmienną środowiskową ANTHROPIC_API_KEY…

**Cause:** AI-assisted translation calls the Anthropic API and needs `ANTHROPIC_API_KEY`. It is an optional variable — the rest of the platform is fully functional without it.

**Fix:** set `ANTHROPIC_API_KEY` in your `.env` or host environment and **restart the server**. The value is read from the process environment, so a running instance will not pick up a newly added key.

<Info>
Auto-translation targets **English, Czech and Slovak**. Asking for anything else returns `Invalid language`; Polish is the source language, and the panel tells you *"Select a language other than Polish."*
</Info>

<AccordionGroup>

<Accordion title="The screens got translated but the posts stayed Polish">
Translation runs as two separate calls: study-level content (consent, instruction, transitions, debriefing, demographic questions, post questions, part labels) and then the posts. The posts call is deliberately non-fatal — if it fails, the study-level translation is still saved and the posts keep their original text. Your server log will show:

```
Posts translation error (non-fatal): …
```

Run **✨ Translate automatically** again, or translate the affected post fields by hand.
</Accordion>

<Accordion title="Translation failed with some other message">
Any error the server does not recognise as an authentication problem is passed through verbatim from the Anthropic SDK (rate limits, network failures, quota). The full error is in the server log next to `Translation error:`. Retrying usually resolves transient failures; nothing is written to the study unless the study-level call succeeds.
</Accordion>

<Accordion title="Do I need a key just to run a study in English?">
No. The key only automates the *first draft* of a translation. Translated text is stored on the study and stays editable by hand, and a study can be authored in another language without ever calling the API.
</Accordion>

</AccordionGroup>

## Images disappear after a redeploy

This one has a counter-intuitive answer.

Post images and avatars are **not** kept as files. When you upload one, the bytes go straight into the database as a BLOB on the post row, and `/uploads/<studyId>/<filename>` streams them back out of the database. There is a fallback that reads from disk, but that exists only for legacy files that predate this design.

So: **images survive a redeploy exactly as well as your database does.** If they vanished, the database file vanished with them.

<Warning>
An ephemeral filesystem is the usual culprit. Most PaaS hosts wipe the container filesystem on every deploy. If `./data/research.db` (or wherever `DATABASE_PATH` points) is not on a persistent volume, you do not just lose images — you lose studies, sessions, reactions, ratings and accounts, and the next boot re-seeds a brand-new `admin` from `ADMIN_PASSWORD`. Losing *everything* rather than only images is the diagnostic signal.
</Warning>

**Fix:** put the database on a persistent volume.

<Tabs>
<Tab title="Railway">
Service → **Volumes** → **Add Volume** with mount path `/app/data`. Redeploy. The default `DATABASE_PATH` (`./data/research.db`) then lands inside the volume.

Add a second volume at `/app/uploads` if the instance is old enough to still hold on-disk images.
</Tab>
<Tab title="Docker Compose">
The bundled `docker-compose.yml` already declares named volumes:

```yaml
volumes:
  - misinfo-data:/app/data
  - misinfo-uploads:/app/uploads
```

If you wrote your own compose file or run `docker run` by hand, mount both paths yourself — the image declares `VOLUME ["/app/data", "/app/uploads"]`.
</Tab>
</Tabs>

<AccordionGroup>

<Accordion title="Old images vanished but new uploads stick">
That instance predates BLOB storage and still had images on disk. On every boot the app scans the uploads directory and copies any file it finds into the matching post's BLOB columns, logging:

```
[image-migration] Loaded 12 image(s) from disk into DB BLOBs
```

The scan is idempotent (it skips posts that already have BLOBs) but it can only migrate files that are still *there* at boot. If the uploads directory was ephemeral, those files are gone and the images must be re-uploaded. Once re-uploaded, they are in the database and the `/app/uploads` volume stops mattering.
</Accordion>

<Accordion title="The upload is rejected">
Two limits apply, both server-side: the file must be **JPEG, PNG or WebP** (anything else fails with `Invalid file type`), and it must be **at most 5 MB**. The panel surfaces the message as an error toast.
</Accordion>

</AccordionGroup>

## A participant gets a 404 on the study link

**Symptom:** the study URL shows *"Badanie nie zostało znalezione lub jest nieaktywne."* ("Study not found or inactive.")

The server resolves participant URLs with exactly two lookups:

| URL | What must be true |
| --- | --- |
| `/study/<slug>` | a study with that slug exists **and `is_active = 1`** |
| `/study/<slug>?preview=1` | a study with that slug exists **and is a builder-mode study** |

Both failures produce the same 404 page, so work through the causes in order:

<Steps>

<Step title="The study is not active">
This is the common one. A new study is created **inactive** on purpose so it cannot be reached before it is finished. Activate it with **Activate** in the **Configurator** (or the **Settings** tab); the study picker in the header marks inactive studies as *inactive*, and the studies list shows an **Inactive** badge.
</Step>

<Step title="The slug is wrong or was changed">
Slugs are unique; creating a study whose slug is taken appends `-2`, `-3` and so on. If you edited **Slug (study URL)** after distributing the link, every previously shared URL now 404s. Treat the slug as frozen once recruitment starts.
</Step>

<Step title="You sent a preview link, or expected one to keep working">
`?preview=1` bypasses the active check, but only for builder-mode studies, and it tags every resulting session as a preview. It is for *your* walkthroughs. Participants must get the plain `/study/<slug>` URL against an active study.
</Step>

<Step title="The link is on a custom domain">
On a bound custom domain almost every path other than the study itself returns a 404 that reads *"Strona nie została znaleziona."* — see the next section.
</Step>

</Steps>

## Rate limiting behind a proxy

Only one endpoint is rate-limited: **admin login**, at 20 attempts per 15 minutes per IP. Participant endpoints are not rate-limited, so a proxy misconfiguration never blocks your sample — it locks out *researchers*.

For that limit to identify callers correctly, the app must know it is behind a proxy. `server.js` declares exactly one trusted hop:

```js
app.set('trust proxy', 1);
```

<AccordionGroup>

<Accordion title="ERR_ERL_UNEXPECTED_X_FORWARDED_FOR in the logs">
The rate limiter saw an `X-Forwarded-For` header while the app did not trust any proxy, so it cannot tell the real client from the proxy and refuses to guess. On a stock checkout this cannot happen — the line above is already there. If you see it, that line was removed or overridden; restore it.
</Accordion>

<Accordion title="Everyone gets locked out of login after a handful of attempts">
Symptom of **more than one proxy hop** in front of the app (for example a CDN in front of your host). With one hop trusted, the address the limiter sees is the intermediate proxy — identical for everybody — so all admins share a single 20-attempts bucket.

The fix is to make the trusted hop count match your actual topology in `server.js`. Count the proxies between the public internet and the app and set the number accordingly.
</Accordion>

<Accordion title="Can I just set trust proxy to true?">
Don't. `true` trusts whatever `X-Forwarded-For` arrives, which lets any client claim any IP and walk straight around the login limiter. Trusting a specific number of hops is what makes the header trustworthy.
</Accordion>

</AccordionGroup>

## Preview sessions where you don't want them

A session row carries an `is_preview` flag. The server sets it to `1` when **either**:

1. the URL carried `?preview=1`, or
2. the browser sent a `missinfo_admin_mode` cookie whose value is `preview`.

The panel sets that cookie to `preview` the moment you sign in (24-hour lifetime, cleared on sign-out). That is what the **My sessions:** switch in the header controls — **🧪 Preview (not counted)** or **📊 Production (counted)**. The dashboard, the analyses and the export all exclude preview sessions by default.

<AccordionGroup>

<Accordion title="My own test runs are counted as real participants">
Your **My sessions:** switch is on **📊 Production (counted)**. The panel warns you when you flip it: *"⚠ Your sessions on study links will count as PRODUCTION. Remember to switch back to Preview mode after testing."* Switch it back to **🧪 Preview (not counted)**, then remove the sessions you already created — see below.
</Accordion>

<Accordion title="Real participants' sessions are missing from the dashboard">
Check whether they used a browser that was signed in to the admin panel — a lab machine or your own laptop handed to a pilot participant. The cookie, not the person, decides the flag. Any session started from a signed-in browser while the switch says Preview is tagged `is_preview = 1` and hidden from the dashboard and export.

The data is not lost. In the **Export** tab, tick the checkbox that adds preview sessions to the export (its label is still shown in Polish: *"Dołącz sesje podglądowe (preview) do eksportu"*) and download again.
</Accordion>

<Accordion title="Preview sessions are cluttering the export">
They are excluded unless you explicitly include them. To remove them for good, use **🗑 Delete preview sessions** in the **Export** tab (the button carries the count in brackets). It deletes both completed and unfinished preview sessions together with their reactions, ratings and answers; production sessions are untouched. The confirmation is blunt about it: *"This operation cannot be undone."*
</Accordion>

<Accordion title="I ticked include-preview but a session still isn't there">
Only **completed** preview sessions are included. Unfinished ones never appear in the export, even with the option on — the panel spells this out in the tooltip: *"Preview sessions that were never finished — they will not appear in the export even with 'Include preview sessions' enabled"*. If there are no completed previews at all you get *"No completed preview sessions to include."*
</Accordion>

</AccordionGroup>

## A custom domain doesn't serve the study

**Custom study domain** (in the **Configurator**) binds one hostname to one study. Requests arriving on that host serve that study at `/` and nothing else — no admin panel, no other studies, no dashboard.

The platform only matches the hostname it receives. Getting traffic to arrive there at all — the DNS record and the TLS certificate — is your DNS and hosting provider's job.

<AccordionGroup>

<Accordion title="The domain shows the admin login instead of the study">
The host → study lookup requires the study to be **active**. If the bound study is inactive the lookup finds nothing, the routing middleware does nothing, and the request falls through to normal routing — where `/` redirects to `/admin`.

So an inactive study on a bound domain does not 404: it exposes your admin login on the domain you gave participants. **Activate the study**, and check the binding again.
</Accordion>

<Accordion title="The domain doesn't match at all">
The stored value must be the **exact bare hostname** — no `https://`, no trailing slash, lowercase, no wildcards. The Configurator normalises what you type (it strips the protocol and any path and lowercases the result) but it intentionally leaves `www.` alone, because only you know whether your CNAME points at the apex or the `www` form. If you bound `example.org` and traffic arrives on `www.example.org`, that is a miss — handle the normalisation in DNS.

The field help says the same thing: *"enter EXACTLY that host name here — without `https://`, without a trailing `/`"*.
</Accordion>

<Accordion title="Some pages on the domain return a Polish 404">
That is the binding working as intended. On a bound host only the study and what it needs are reachable: `/` (the bound study), that study's own `/study/<slug>` URL, `/api/*`, `/uploads/*`, and the static assets under `/css/`, `/js/`, `/locales/`, `/favicon` plus the eye-tracking model path. Everything else — including `/admin` — returns *"Strona nie została znaleziona."*

Use your installation's original address for the admin panel. Recruitment links keep working: the query string is preserved, so `https://study.example.org/?res_id=ABC123` still lands the panel ID on the session.
</Accordion>

<Accordion title="Two studies are bound to the same hostname">
Nothing stops you from saving the same hostname on two studies — uniqueness is not enforced in the schema, only at lookup time, where the first matching active study wins. Which one that is, is not something you should rely on. Bind one host to one study, and clear the field on the other (empty = the study is reachable only via `/study/<slug>` on the app's main address).
</Accordion>

</AccordionGroup>

## When nothing above fits

<Card title="Read the server log first">
The platform reports its real problems on stdout, not in the UI: the fatal `JWT_SECRET` line, the `[accounts]` seed messages, `[image-migration]` counts, and the full text of any failed translation. Your host's log view (or `docker compose logs -f app`) answers more questions than the panel does.
</Card>

If the behaviour still looks wrong after that, it may be a bug rather than a configuration problem — the platform is open source (MIT), so the code is the final documentation. Report it at [github.com/lowcydizajnu/missinfo-research](https://github.com/lowcydizajnu/missinfo-research) with the relevant log lines and the steps that produced it.
