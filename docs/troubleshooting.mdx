title: "Troubleshooting common Misinfo Research setup issues"
sidebarTitle: "Troubleshooting"
description: "Step-by-step fixes for the most common problems in Misinfo Research: server startup, login, 404 links, missing images, data export, and more."
---

Most issues with Misinfo Research fall into one of a small number of categories: missing environment variables, study configuration oversights, or mismatches between what the server expects and what it receives. Work through the relevant accordion below — each entry follows a **Symptom → Cause → Fix** structure so you can confirm you have the right issue before making any changes.

<AccordionGroup>

  <Accordion title="Server refuses to start">
    **Symptom:** The Node.js process exits immediately on startup, often with an error referencing a missing secret or a configuration problem.

    **Cause:** The `JWT_SECRET` environment variable is not set. Misinfo Research requires this variable to sign and verify admin authentication tokens. Without it, the server will not start, as running without a secret would make all admin sessions cryptographically insecure.

    **Fix:**

    <Steps>
      <Step title="Open your environment configuration">
        If you are running locally, open (or create) the `.env` file in the project root. If you are running on a platform like Railway or Render, open the environment variables panel for your service.
      </Step>
      <Step title="Add the JWT_SECRET variable">
        Set `JWT_SECRET` to any long, random string. A minimum of 48 bytes (96 hex characters) is recommended. You can generate one with:
        ```bash
        node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
        ```
      </Step>
      <Step title="Restart the server">
        Save your changes and restart the process. The server should now start without errors. Check the logs to confirm there are no other missing required variables.
      </Step>
    </Steps>

    <Warning>
      Never commit your `.env` file to version control. Add it to `.gitignore` before your first commit.
    </Warning>
  </Accordion>

  <Accordion title="Can't log in or locked out of the admin panel">
    **Symptom:** Navigating to `/admin` and entering your credentials returns an "Invalid password" error, or you cannot remember the password and have no way to reset it through the UI.

    **Cause:** The `ADMIN_PASSWORD` environment variable is used to seed the admin account **on first boot only** — specifically, when no admin user exists in the database yet. If the variable was not set before the first run, or if you have changed it after the initial seed, the stored password will not match what you are entering.

    **Fix:**

    <Steps>
      <Step title="Check whether an admin user already exists">
        If the database was already seeded (i.e., the server has been started at least once before), simply changing `ADMIN_PASSWORD` in `.env` and restarting will **not** update the password. The seed only runs when there is no admin in the database.
      </Step>
      <Step title="Reset the password via the server console">
        Connect to your server and run the password-reset script described in the project README to update the stored password hash directly.
      </Step>
      <Step title="Alternatively, wipe and re-seed (development only)">
        If you are in a development environment and have no data you need to keep, you can delete the SQLite database file and restart the server with `ADMIN_PASSWORD` set. The server will create a fresh database and seed the admin account with the new password.
      </Step>
    </Steps>

    <Note>
      In production, always use the console reset path. Wiping the database destroys all study data and cannot be undone.
    </Note>
  </Accordion>

  <Accordion title="Participant link returns a 404 error">
    **Symptom:** A participant (or you, during testing) opens the `/study/<slug>` URL and sees a 404 or "Study not found" page.

    **Cause:** There are two common causes:
    1. The study has not been **activated**. Studies are inactive by default and do not accept participants until you explicitly activate them.
    2. The **slug in the URL is wrong** — either a typo or the slug was automatically suffixed (e.g., `-2`) because of a conflict with an existing study name.

    **Fix:**

    <Steps>
      <Step title="Confirm the study is active">
        Log in to `/admin`, open the study, and check its status. If it shows as inactive or draft, toggle it to active and save.
      </Step>
      <Step title="Confirm the correct slug">
        In the study Settings tab, check the **Study slug** field. Copy the slug exactly as shown and construct the link as `/study/<slug>`. If the slug ends in `-2` or `-3`, the link you shared may be pointing to the original un-suffixed slug.
      </Step>
      <Step title="Rename the slug before resharing">
        If the auto-suffixed slug is the problem, edit it to something clean and memorable in Settings, then reshare the updated link with participants.
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="My test sessions are appearing in production data">
    **Symptom:** When you view the dashboard or export data, you can see sessions that you ran yourself during testing — they are inflating your completion counts or skewing response distributions.

    **Cause:** When you opened the live study link, the **"My sessions"** toggle in the participant view was set to **Production** rather than **Preview**. This caused your session to be recorded as a real participant session instead of a preview session.

    **Fix:**

    <Steps>
      <Step title="Open the Export tab for your study">
        In the admin panel, navigate to the affected study and click the **Export** tab.
      </Step>
      <Step title="Identify and delete your test sessions">
        Preview sessions and production sessions are listed separately. Locate your test sessions — you can usually identify them by timestamp or by the UUID if you noted it — and delete them individually. You can also delete all preview sessions at once if none of them should be in the dataset.
      </Step>
      <Step title="Set the toggle correctly for future tests">
        Before opening your own study link for any future testing, confirm the toggle is set to **Preview** so your sessions are automatically excluded from exports and dashboard statistics.
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="Post images are not loading">
    **Symptom:** Post cards display broken image icons instead of the images you uploaded. The rest of the study (text, buttons, Likert scales) loads normally.

    **Cause:** Uploaded files are stored in the `./uploads/` directory on the server by default. In containerised or cloud deployments, this directory may not be persisted across restarts, or the `UPLOADS_PATH` environment variable may be pointing to a location that the server process cannot read.

    **Fix:**

    <Steps>
      <Step title="Check the UPLOADS_PATH variable">
        Confirm that `UPLOADS_PATH` in your environment points to the directory where images were originally uploaded. If the variable is unset, the server defaults to `./uploads/` relative to the project root.
      </Step>
      <Step title="Verify the directory exists and is readable">
        SSH into your server (or open a shell in your container) and confirm the uploads directory exists and contains your image files. If the directory is empty, the images were not persisted.
      </Step>
      <Step title="Mount a persistent volume (Docker / Railway)">
        If you are running in Docker or on Railway, ensure you have a persistent volume mounted to the uploads path. Without a volume, files written inside the container are lost on every restart. After mounting, re-upload any images that were lost.
      </Step>
    </Steps>

    <Warning>
      Do not rely on ephemeral container storage for uploads in production. Always configure a persistent volume or object storage before going live with a study.
    </Warning>
  </Accordion>

  <Accordion title="Condition cells are filling unevenly">
    **Symptom:** After collecting a number of sessions, you notice that one condition has significantly more participants than another, even though you expected balanced cells.

    **Cause:** **Builder mode studies use uniform random assignment**, which means each participant is independently assigned to a condition with equal probability. Over small samples, this can produce noticeable imbalance by chance — just as flipping a fair coin ten times does not guarantee five heads.

    **Classic mode studies** use **permuted block randomisation**, which does guarantee balanced cells within each block and results in more even condition sizes at every point during data collection.

    **Fix:**

    <Steps>
      <Step title="Assess whether the imbalance is within acceptable bounds">
        For most studies, minor imbalance from random assignment is statistically unproblematic and should be accounted for in your analysis (e.g., by including condition as a covariate). A difference of a few sessions between cells is expected and not a sign of a bug.
      </Step>
      <Step title="Switch to classic mode if balance is critical">
        If your protocol requires guaranteed equal cell sizes at every interim check, consider using a classic study, which uses permuted block randomisation. Note that classic and builder studies have different feature sets, so review the FAQ entry on classic vs builder mode before switching.
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="Export shows fewer sessions than the dashboard">
    **Symptom:** The number of sessions in your downloaded export file is lower than the total session count shown on the study dashboard.

    **Cause:** The data export includes **only completed sessions** — participants who reached the end of the study and submitted their responses. Participants who started the study but dropped out before completing it appear in the dashboard's session and completion-rate counts, but their partial records are excluded from the export.

    **Fix:**

    <Steps>
      <Step title="Check completion rate on the dashboard">
        The dashboard shows both total sessions started and the completion rate. The difference between those two numbers is your drop-out count. This is expected behaviour, not a data loss bug.
      </Step>
      <Step title="Investigate the drop-off stage">
        The dashboard also shows which stage of the study participants were at when they dropped out. If a particular stage has high abandonment, consider whether the instructions are clear, whether the page is too long, or whether there is a rendering issue at that point.
      </Step>
      <Step title="Do not expect partial data in the export">
        If you need to analyse drop-out patterns beyond what the dashboard provides, you would need to access the SQLite database directly and query the sessions table for incomplete records.
      </Step>
    </Steps>
  </Accordion>

  <Accordion title="AI translation button is not visible in the builder">
    **Symptom:** You expected to see a translate or AI-assist button in the study builder for post content or interface strings, but no such button appears.

    **Cause:** The AI translation feature is only available when a valid `ANTHROPIC_API_KEY` is configured on the server instance. If the environment variable is missing or empty, the translation UI is hidden entirely.

    **Fix:**

    <Steps>
      <Step title="Obtain an Anthropic API key">
        Sign up for API access at [anthropic.com](https://www.anthropic.com) and generate an API key from your account dashboard.
      </Step>
      <Step title="Set the environment variable">
        Add `ANTHROPIC_API_KEY=<your-key>` to your `.env` file or your hosting platform's environment variables panel.
      </Step>
      <Step title="Restart the server">
        Restart the Node.js process to pick up the new variable. The translation button should now appear in the builder without any further configuration.
      </Step>
    </Steps>

    <Note>
      All other platform features — study building, data collection, and export — work without this key. It is only required for AI-assisted translation.
    </Note>
  </Accordion>

  <Accordion title="New study has an unexpected slug suffix (-2, -3, etc.)">
    **Symptom:** You created a new study and noticed that its auto-generated slug ends in `-2`, `-3`, or another number — for example, `vaccine-attitudes-2` instead of `vaccine-attitudes`.

    **Cause:** Misinfo Research requires every study slug to be unique across the instance. When you create a study whose title would generate a slug that already exists (from a previous study, even a deleted one that left a slug behind), the platform automatically appends a numeric suffix to avoid a conflict.

    **Fix:**

    <Steps>
      <Step title="Rename the slug before sharing any links">
        Open the study's **Settings** tab and edit the slug field to the value you want. Choose something unique, descriptive, and URL-safe (lowercase letters, numbers, and hyphens only).
      </Step>
      <Step title="Save and verify">
        Save the settings and confirm the participant link now reads `/study/<your-chosen-slug>`. Test it in your browser before distributing it to participants.
      </Step>
    </Steps>

    <Warning>
      If you have already shared the suffixed link with any participants, do **not** rename the slug — doing so will break the link for anyone who has bookmarked or received it. Only rename before the study goes live.
    </Warning>
  </Accordion>

</AccordionGroup>
