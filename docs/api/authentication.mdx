Admin endpoints in the Misinfo Research API are protected with JSON Web Tokens (JWT). You obtain a token by posting your credentials to the login endpoint, then include that token as a `Bearer` header on every subsequent admin request. Participant endpoints — the session, reaction, and rating endpoints — require no authentication at all.

## Getting a Token

Send a `POST` request to `/api/admin/login` with your username and password. On success you receive a token along with your username and role.

<CodeGroup>
```bash Login request
curl -X POST https://your-host/api/admin/login \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "password": "s3cr3t"}'
```

```json Response
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "role": "researcher",
  "username": "alice"
}
```
</CodeGroup>

Store the returned `token` value securely. You will attach it to all admin API calls.

### Request Parameters

<ParamField body="username" type="string" required>
  The account username registered on your Misinfo Research instance.
</ParamField>

<ParamField body="password" type="string" required>
  The account password.
</ParamField>

### Response Fields

<ResponseField name="token" type="string">
  A signed JWT. Valid for **24 hours** from the time of issue.
</ResponseField>

<ResponseField name="role" type="string">
  Either `"admin"` or `"researcher"`. Controls which operations you are permitted to perform.
</ResponseField>

<ResponseField name="username" type="string">
  The username of the authenticated account, echoed back for confirmation.
</ResponseField>

## Using the Token

Include the token in the `Authorization` header of every admin request using the `Bearer` scheme:

```http
Authorization: Bearer <token>
```

<CodeGroup>
```bash Authenticated request
curl https://your-host/api/admin/studies \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```
</CodeGroup>

<Warning>
Tokens expire after **24 hours**. When a token expires, the API returns `401 Unauthorized`. Repeat the login flow to obtain a fresh token. Do not embed tokens in client-side code or commit them to source control.
</Warning>

## Token Lifetime

| Property | Value |
|---|---|
| Expiry | 24 hours from issue |
| Algorithm | JWT (signed server-side) |
| Transport | `Authorization: Bearer` header only |

There is no refresh endpoint. When your token expires, log in again with your credentials to obtain a new one.

## Rate Limiting on Login

To protect against brute-force attacks, the login endpoint is rate-limited:

<Note>
You may make a maximum of **20 login attempts per 15-minute window** per IP address. Exceeding this limit returns `429 Too Many Requests`. Wait for the window to reset before retrying.
</Note>

## Roles and Permissions

Every authenticated user has one of two roles returned at login time.

### `admin`

Admins have unrestricted access to all studies and all admin operations across the entire instance, regardless of who created the study.

### `researcher`

Researchers can access their own studies and any studies they have been added to as a collaborator. Attempting to access another user's study returns `404` (not `403`) to prevent enumeration. Attempting an operation that is strictly admin-only returns `403 Forbidden`.

| Operation | `admin` | `researcher` |
|---|---|---|
| View own studies | ✅ | ✅ |
| View all studies on instance | ✅ | ❌ |
| Create / edit / delete studies | ✅ | ✅ (own only) |
| Access collaborated studies | ✅ | ✅ |
| Admin-only operations | ✅ | ❌ (`403`) |

## Authentication Errors

| Status | Meaning | When it occurs |
|---|---|---|
| `401` | Unauthorized | `Authorization` header missing, token malformed, or token expired |
| `403` | Forbidden | Token is valid but the role does not permit the requested operation |

<Tip>
If you receive a `401` on a request that previously worked, your token has likely expired. Re-authenticate via `POST /api/admin/login` and retry with the new token.
</Tip>
