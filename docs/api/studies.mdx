The Studies and Posts API gives you full programmatic control over the content and configuration of your misinformation experiments. All endpoints in this section require a valid JWT Bearer token — see [Authentication](/api/authentication) for how to obtain one.

---

## Studies

### List Studies

```http
GET /api/admin/studies
```

Returns an array of study objects visible to the authenticated user. Admins see all studies on the instance; researchers see only studies they own or collaborate on.

<CodeGroup>
```bash Request
curl https://your-host/api/admin/studies \
  -H "Authorization: Bearer <token>"
```

```json Response
[
  {
    "id": 1,
    "name": "Vaccine Misinformation Study",
    "slug": "vaccine-misinfo",
    "is_active": true,
    "language": "en",
    "created_at": "2024-03-01T10:00:00Z"
  }
]
```
</CodeGroup>

### Create a Study

```http
POST /api/admin/studies
```

Creates a new study. The slug is used in participant-facing URLs; if omitted, one is generated automatically from the study name.

#### Request Body

<ParamField body="name" type="string" required>
  Human-readable name for the study, shown in the admin dashboard.
</ParamField>

<ParamField body="slug" type="string">
  URL-safe identifier used in participant links (e.g. `vaccine-misinfo`). Must be unique across the instance. Auto-generated from `name` if omitted.
</ParamField>

#### Response

<ResponseField name="id" type="number">
  Unique integer ID of the newly created study.
</ResponseField>

<ResponseField name="slug" type="string">
  The slug assigned to the study, either as supplied or auto-generated.
</ResponseField>

<CodeGroup>
```bash Request
curl -X POST https://your-host/api/admin/studies \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"name": "Climate Denial Study", "slug": "climate-denial"}'
```

```json Response
{
  "id": 2,
  "slug": "climate-denial",
  "name": "Climate Denial Study",
  "is_active": false
}
```
</CodeGroup>

### Update a Study

```http
PATCH /api/admin/studies/:id
```

Partially updates a study. Send only the fields you want to change; all other fields remain unchanged.

<Note>
This endpoint returns the **full updated study object**, not just `{"ok": true}`. You can use the returned object to refresh your local state without a separate GET request.
</Note>

#### Path Parameters

<ParamField path="id" type="number" required>
  The integer ID of the study to update.
</ParamField>

#### Request Body

<ParamField body="name" type="string">
  Updated display name for the study.
</ParamField>

<ParamField body="slug" type="string">
  Updated URL slug. Must remain unique across the instance.
</ParamField>

<ParamField body="participant_title" type="string">
  Title shown to participants at the top of the study interface.
</ParamField>

<ParamField body="description" type="string">
  Internal description visible only in the admin dashboard.
</ParamField>

<ParamField body="language" type="string">
  BCP 47 language code (e.g. `"en"`, `"de"`, `"fr"`) controlling the study's locale strings.
</ParamField>

<ParamField body="contact_email" type="string">
  Contact email shown to participants on the debrief screen.
</ParamField>

<ParamField body="institution" type="string">
  Institution name shown in participant-facing text.
</ParamField>

<ParamField body="is_active" type="boolean">
  When `false`, the study is closed to new participants. Accessing a study with `is_active: false` via the participant API returns `404`.
</ParamField>

<ParamField body="show_metrics" type="boolean">
  Whether engagement metrics (likes, shares, etc.) are shown to participants.
</ParamField>

#### Response

The full updated study object is returned. Key fields include those sent in the request plus all unchanged study fields.

<CodeGroup>
```bash Request
curl -X PATCH https://your-host/api/admin/studies/2 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"is_active": true, "contact_email": "pi@university.edu"}'
```

```json Response
{
  "id": 2,
  "name": "Climate Denial Study",
  "slug": "climate-denial",
  "is_active": true,
  "contact_email": "pi@university.edu",
  "language": "en",
  "created_at": "2024-03-01T10:00:00Z"
}
```
</CodeGroup>

### Delete a Study

```http
DELETE /api/admin/studies/:id
```

Permanently deletes a study. This also deletes all posts, participant sessions, reactions, ratings, and any other data associated with the study.

<Warning>
Deletion is **irreversible**. All posts, sessions, and collected data are removed permanently. Export your data first using `GET /api/admin/export/:studyId` if you need to preserve it.
</Warning>

#### Path Parameters

<ParamField path="id" type="number" required>
  The integer ID of the study to delete.
</ParamField>

#### Request Body

<ParamField body="confirm" type="string" required>
  Must be the exact string `"DELETE"`. This confirmation field is required to prevent accidental deletions. Omitting it or sending any other value returns `400 Bad Request`.
</ParamField>

#### Response

<ResponseField name="ok" type="boolean">
  `true` on success.
</ResponseField>

<CodeGroup>
```bash Request
curl -X DELETE https://your-host/api/admin/studies/2 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"confirm": "DELETE"}'
```

```json Response
{ "ok": true }
```
</CodeGroup>

### Duplicate a Study

```http
POST /api/admin/studies/:id/duplicate
```

Creates a copy of the study with all its posts. The duplicate has no participant sessions. Use this to create variants of an existing study without starting from scratch.

#### Path Parameters

<ParamField path="id" type="number" required>
  The integer ID of the study to duplicate.
</ParamField>

#### Response

The full study object for the newly created duplicate is returned — the same shape as the `GET /api/admin/studies` list entries. Key fields include:

<ResponseField name="id" type="number">
  Integer ID of the newly created duplicate study.
</ResponseField>

<ResponseField name="slug" type="string">
  Auto-generated slug for the duplicate, derived from the original slug with a numeric copy suffix.
</ResponseField>

<ResponseField name="name" type="string">
  Display name of the duplicate, derived from the original name with a copy suffix.
</ResponseField>

<CodeGroup>
```bash Request
curl -X POST https://your-host/api/admin/studies/2/duplicate \
  -H "Authorization: Bearer <token>"
```

```json Response
{
  "id": 3,
  "slug": "climate-denial-copy",
  "name": "Climate Denial Study (kopia)",
  "is_active": false,
  "language": "en",
  "created_at": "2024-04-01T12:00:00Z"
}
```
</CodeGroup>

### Create a Post

```http
POST /api/admin/posts
```

Creates a new post and appends it to the specified study. Posts support an A/B variant system: `headline_a`/`content_a` and `headline_b`/`content_b` allow you to randomize which version a participant sees.

#### Request Body

<ParamField body="study_id" type="number" required>
  ID of the study this post belongs to.
</ParamField>

<ParamField body="source_name" type="string">
  Display name of the post's simulated source (e.g. `"The Daily News"`).
</ParamField>

<ParamField body="source_handle" type="string">
  Social-media-style handle for the source (e.g. `"@dailynews"`).
</ParamField>

<ParamField body="time_ago" type="string">
  Relative timestamp string displayed on the post (e.g. `"2 hours ago"`).
</ParamField>

<ParamField body="topic" type="string">
  Topic label associated with the post (e.g. `"vaccines"`).
</ParamField>

<ParamField body="emoji" type="string">
  Single emoji displayed alongside the topic label.
</ParamField>

<ParamField body="hide_topic" type="boolean">
  When `true`, the topic label and emoji are hidden from participants.
</ParamField>

<ParamField body="headline_a" type="string">
  Headline text for variant A of the post.
</ParamField>

<ParamField body="content_a" type="string">
  Body text for variant A of the post.
</ParamField>

<ParamField body="headline_b" type="string">
  Headline text for variant B of the post. If omitted, variant A is always shown.
</ParamField>

<ParamField body="content_b" type="string">
  Body text for variant B of the post.
</ParamField>

<ParamField body="is_true" type="boolean">
  Ground-truth label indicating whether the post content is accurate. Used in analysis; not shown to participants.
</ParamField>

<ParamField body="is_active" type="boolean">
  When `false`, the post is excluded from participant sessions.
</ParamField>

<ParamField body="base_likes" type="number">
  Starting like count displayed to participants before any session interactions are added.
</ParamField>

<ParamField body="base_shares" type="number">
  Starting share count displayed to participants.
</ParamField>

<ParamField body="base_dislikes" type="number">
  Starting dislike count displayed to participants.
</ParamField>

<ParamField body="base_flags" type="number">
  Starting flag count displayed to participants.
</ParamField>

<ParamField body="part_id" type="number">
  ID of the study part this post belongs to, for multi-part studies.
</ParamField>

#### Response

The full post object for the newly created post is returned. Key fields include:

<ResponseField name="id" type="number">
  Integer ID of the newly created post.
</ResponseField>

<ResponseField name="study_id" type="number">
  ID of the study this post belongs to.
</ResponseField>

<ResponseField name="order_index" type="number">
  Position of this post within the study's post list (appended at the end on creation).
</ResponseField>

### Update a Post

```http
PATCH /api/admin/posts/:id
```

Partially updates a post. Send only the fields you want to change.

#### Path Parameters

<ParamField path="id" type="number" required>
  Integer ID of the post to update.
</ParamField>

#### Request Body

All fields from the `POST /api/admin/posts` body are accepted as optional partial updates.

#### Response

The full updated post object is returned — the same shape as the `GET /api/admin/studies/:id/posts` list entries. You can use this to refresh your local post state without a separate GET request.

### Reorder a Post

```http
PATCH /api/admin/posts/:id/reorder
```

Moves a post one position up or down within its study's post list.

#### Path Parameters

<ParamField path="id" type="number" required>
  Integer ID of the post to reorder.
</ParamField>

#### Request Body

<ParamField body="direction" type="string" required>
  Either `"up"` to move the post earlier in the list, or `"down"` to move it later.
</ParamField>

#### Response

<ResponseField name="ok" type="boolean">
  `true` on success.
</ResponseField>

### Upload a Post Image

```http
POST /api/admin/posts/:id/image
```

Attaches an image to a post. The request must be sent as `multipart/form-data`.

#### Path Parameters

<ParamField path="id" type="number" required>
  Integer ID of the post to attach an image to.
</ParamField>

#### Request Body (multipart/form-data)

<ParamField body="image" type="file" required>
  The image file to upload. Accepted formats: `jpg`, `png`, `webp`. Maximum file size: **5 MB**.
</ParamField>

#### Response

<ResponseField name="image_url" type="string">
  Root-relative URL of the uploaded image as served by your host (e.g. `/uploads/2/10_abc123.png`). Prepend your base URL to construct the full absolute URL.
</ResponseField>

<ResponseField name="filename" type="string">
  The server-assigned filename for the uploaded image. Each upload generates a unique timestamped filename to prevent stale CDN or browser cache hits.
</ResponseField>

<ResponseField name="col" type="string">
  Internal database column name the image was stored in. You can safely ignore this field in most integrations.
</ResponseField>

<CodeGroup>
```bash Request
curl -X POST https://your-host/api/admin/posts/10/image \
  -H "Authorization: Bearer <token>" \
  -F "image=@/path/to/image.png"
```

```json Response
{
  "image_url": "/uploads/2/10_abc123.png",
  "filename": "10_abc123.png",
  "col": "image_path"
}
```
</CodeGroup>

<Note>
Uploading a new image replaces any previously uploaded image for the same post. The old file is removed from storage.
</Note>

### Export Session Data (XLSX)

```http
GET /api/admin/export/:studyId
```

Downloads all session and reaction data for a study as an Excel workbook (`.xlsx`). This is the primary export format and produces a multi-sheet workbook with formatted column headers.

#### Path Parameters

<ParamField path="studyId" type="number" required>
  Integer ID of the study to export.
</ParamField>

#### Response

Returns an `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` file download. The `Content-Disposition` header includes a suggested filename in the format `<slug>_<date>.xlsx`.

<CodeGroup>
```bash Download XLSX
curl "https://your-host/api/admin/export/2" \
  -H "Authorization: Bearer <token>" \
  --output study-2-data.xlsx
```
</CodeGroup>

---

### Export Session Data (CSV)

```http
GET /api/admin/export/:studyId/csv
```

Downloads the primary data sheet for a study as a UTF-8 CSV file. Use this when you want to pipe data directly into R, Python, or another analysis tool that works with plain text.

#### Path Parameters

<ParamField path="studyId" type="number" required>
  Integer ID of the study to export.
</ParamField>

#### Response

Returns a `text/csv; charset=utf-8` file download with a UTF-8 BOM prepended so Excel opens the file correctly without a character-encoding prompt. The `Content-Disposition` header includes a suggested filename in the format `<slug>_<date>.csv`.

<CodeGroup>
```bash Download CSV
curl "https://your-host/api/admin/export/2/csv" \
  -H "Authorization: Bearer <token>" \
  --output study-2-data.csv
```
</CodeGroup>

<Tip>
Use `GET /api/admin/export/:studyId` (XLSX) when you need multiple worksheets, formatted column headers, or Excel compatibility. Use `GET /api/admin/export/:studyId/csv` when you want to feed data directly into a script or analysis pipeline.
</Tip>
