Upgrading Misinfo Research is straightforward regardless of how you deployed it. The application runs any required database schema migrations automatically on startup, so you never need to write or execute SQL by hand. Follow the instructions below for your deployment method.

## Before you upgrade

It is good practice to take a backup before upgrading, so you have a rollback point if something goes wrong. See the [Backup guide](/hosting/backup) for instructions.

## Node.js upgrade

<Steps>
  <Step title="Pull the latest code">
    From inside your project directory, fetch and apply the latest commits:

    ```bash
    git pull
    ```
  </Step>

  <Step title="Install updated dependencies">
    Dependencies may have been added, removed, or upgraded in the new version. Run `npm install` to synchronise your local `node_modules` with the updated `package.json`:

    ```bash
    npm install
    ```
  </Step>

  <Step title="Restart the server">
    ```bash
    npm start
    ```

    On startup the application automatically detects any pending schema migrations and applies them to the database before accepting requests. You will see migration activity in the server logs if changes were made.
  </Step>
</Steps>

<Note>
  If you are running the server as a background service (e.g. via `systemd` or `pm2`), reload or restart the service instead of running `npm start` directly. For example: `pm2 restart misinfo` or `systemctl restart misinfo`.
</Note>

## Docker Compose upgrade

<Steps>
  <Step title="Pull the latest image">
    Fetch the updated container image (and any updated images for other services):

    ```bash
    docker compose pull
    ```
  </Step>

  <Step title="Recreate the containers">
    Start new containers from the updated images. Running containers are replaced with zero manual cleanup:

    ```bash
    docker compose up -d
    ```

    Docker Compose replaces only the containers whose images have changed, leaving your named volumes (`misinfo-data` and `misinfo-uploads`) untouched.
  </Step>
</Steps>

<Tip>
  You can combine both steps into a single command: `docker compose pull && docker compose up -d`. This is convenient to run from a scheduled job or a deployment script.
</Tip>

## Railway upgrade

Railway redeploys your service automatically whenever changes are pushed to the connected GitHub repository.

<Steps>
  <Step title="Push your changes to GitHub">
    If you are working from a fork or a local checkout, push the latest commits to the branch Railway is tracking:

    ```bash
    git push
    ```

    Railway detects the push event via a webhook and immediately queues a new build and deploy.
  </Step>

  <Step title="Monitor the deploy">
    Open the Railway dashboard and navigate to your service. The **Deployments** tab shows the build log in real time. The new version goes live as soon as the deploy completes successfully.
  </Step>
</Steps>

<Note>
  Railway's deploy process does not affect your persistent volumes. The database and uploads on `/app/data` and `/app/uploads` are preserved across all redeploys.
</Note>

## Automatic schema migrations

Misinfo Research manages its own database schema. Every time the application starts, it checks whether the database is up to date and applies any pending migrations automatically. You do not need to:

- Run any SQL commands manually
- Use a separate migration tool
- Reset or recreate the database

This applies to all deployment methods: Node.js, Docker Compose, and Railway.

<Warning>
  Automatic migrations modify the database schema. Always take a backup before upgrading a production instance so you can restore the previous state if an unexpected issue occurs.
</Warning>
