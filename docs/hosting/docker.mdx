Docker Compose gives you a consistent, isolated environment without installing Node.js on your host machine. It manages the container lifecycle and mounts named volumes so your data survives container rebuilds.

## Prerequisites

- **Docker** — version 20.10 or later; check with `docker -v`
- **Docker Compose** — included in Docker Desktop; on Linux install the `docker-compose-plugin`; check with `docker compose version`
- **git** — to clone the repository

## Setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/lowcydizajnu/missinfo-research.git
    cd missinfo-research
    ```
  </Step>

  <Step title="Create your environment file">
    Copy the included template to create a `.env` file that Docker Compose reads automatically:

    ```bash
    cp .env.example .env
    ```
  </Step>

  <Step title="Set required variables">
    Open `.env` in a text editor and provide values for the two required variables.

    Generate a secure `JWT_SECRET`:

    ```bash
    node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
    ```

    Without Node.js locally, any method that produces 48+ random bytes as a hex string works (e.g. `openssl rand -hex 48`).

    Then set both values in your `.env`:

    ```env
    JWT_SECRET=your_generated_secret_here
    ADMIN_PASSWORD=your_strong_password_here
    ```

    <Warning>
      `JWT_SECRET` is required. The application refuses to start without it, and Docker Compose keeps restarting the container until it is set.
    </Warning>
  </Step>

  <Step title="Start the containers">
    Launch the stack in detached mode so it runs in the background:

    ```bash
    docker compose up -d
    ```

    Docker Compose pulls the required images (if not already cached), builds the application image, and starts the container. The admin panel is available at `http://localhost:3000/admin` once the container is healthy.
  </Step>
</Steps>

## Persistent volumes

Docker Compose creates two named volumes automatically. Docker stores them on your host, and they persist across container rebuilds, image updates, and `docker compose down` calls unless you explicitly delete them.

| Volume name | Mount path inside container | Contents |
|---|---|---|
| `misinfo-data` | `/app/data` | SQLite database (`research.db`) |
| `misinfo-uploads` | `/app/uploads` | User-uploaded images |

<Note>
  Because data lives in named volumes rather than bind mounts, there are no file-permission issues to manage between your host user and the container user.
</Note>

## Managing the container

**Check that the container is running:**

```bash
docker compose ps
```

The service is listed with a status of `Up` or `running`.

**View live logs:**

```bash
docker compose logs -f
```

Press `Ctrl + C` to stop following logs without stopping the container.

**Stop the containers:**

```bash
docker compose down
```

This stops and removes the containers but leaves the named volumes, so your data is safe.

**Stop and remove volumes (destructive):**

```bash
docker compose down -v
```

<Warning>
  The `-v` flag permanently deletes the named volumes and all data inside them. Use it only when you intend to wipe the instance entirely.
</Warning>

## Accessing the admin panel

Once the container is running, open:

```
http://localhost:3000/admin
```

Log in with username `admin` and the password you set in `ADMIN_PASSWORD`.

<Tip>
  To expose the application on a different host port, add a `ports` override or set `PORT` in your `.env` file before running `docker compose up -d`.
</Tip>
