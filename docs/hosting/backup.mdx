All persistent state in Misinfo Research lives in two places: a SQLite database file and an uploads directory. Backing up your instance means copying both. This page explains where they are, how to take a consistent snapshot, and how to restore from a backup.

## What to back up

| Data | Default path | Override variable |
|---|---|---|
| SQLite database | `./data/research.db` | `DATABASE_PATH` |
| Uploaded images | `./uploads/` | `UPLOADS_PATH` |

If you have set `DATABASE_PATH` or `UPLOADS_PATH` in your `.env`, use those paths instead of the defaults shown above.

## Cold backup (server stopped)

Stopping the server before copying ensures you capture a fully consistent snapshot with no in-flight writes.

<Steps>
  <Step title="Stop the server">
    <Tabs>
      <Tab title="Node.js">
        Press `Ctrl + C` in the terminal where the server is running, or send `SIGTERM` to the process.
      </Tab>
      <Tab title="Docker">
        ```bash
        docker compose down
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Copy the database and uploads">
    ```bash
    cp ./data/research.db /path/to/your/backup/research.db
    cp -r ./uploads/ /path/to/your/backup/uploads/
    ```

    Adjust the source paths if you use `DATABASE_PATH` or `UPLOADS_PATH`.
  </Step>

  <Step title="Restart the server">
    <Tabs>
      <Tab title="Node.js">
        ```bash
        npm start
        ```
      </Tab>
      <Tab title="Docker">
        ```bash
        docker compose up -d
        ```
      </Tab>
    </Tabs>
  </Step>
</Steps>

## Hot backup (server running)

If you cannot afford downtime, SQLite's built-in `.backup` command writes a consistent copy of the database while the server is live. This avoids the need to stop the application.

```bash
sqlite3 ./data/research.db ".backup /path/to/your/backup/research.db"
```

<Note>
  The `.backup` command is safe to run while the application is writing to the database. SQLite's backup API takes a read lock only for the brief moment it copies each page, so normal operation is not blocked.
</Note>

After the database backup completes, copy the uploads directory:

```bash
cp -r ./uploads/ /path/to/your/backup/uploads/
```

<Info>
  The hot backup guarantees consistency for the database. If a file upload occurs between the database backup and the uploads copy, the backup may reference an image that is not yet included. For a perfectly consistent snapshot, use the cold backup method.
</Info>

## Docker volume backup

When running with Docker Compose, data lives in named volumes (`misinfo-data` and `misinfo-uploads`) rather than the host filesystem. Use a temporary container to stream the volume contents to a tar archive.

**Back up the database volume:**

```bash
docker run --rm \
  -v misinfo-data:/data \
  -v "$(pwd)/backups":/backup \
  alpine tar czf /backup/misinfo-data.tar.gz -C /data .
```

**Back up the uploads volume:**

```bash
docker run --rm \
  -v misinfo-uploads:/uploads \
  -v "$(pwd)/backups":/backup \
  alpine tar czf /backup/misinfo-uploads.tar.gz -C /uploads .
```

This writes compressed archives to a `backups/` directory on your host. Run these commands while the application stack is stopped for a cold backup, or while it is running for a best-effort hot backup.

## Restoring from a backup

<Steps>
  <Step title="Stop the server">
    <Tabs>
      <Tab title="Node.js">
        Stop the running process (`Ctrl + C` or `SIGTERM`).
      </Tab>
      <Tab title="Docker">
        ```bash
        docker compose down
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Restore the database and uploads">
    <Tabs>
      <Tab title="Node.js / bare files">
        Copy your backed-up files back to their original locations:

        ```bash
        cp /path/to/your/backup/research.db ./data/research.db
        cp -r /path/to/your/backup/uploads/ ./uploads/
        ```
      </Tab>
      <Tab title="Docker volumes">
        Restore the database volume:

        ```bash
        docker run --rm \
          -v misinfo-data:/data \
          -v "$(pwd)/backups":/backup \
          alpine tar xzf /backup/misinfo-data.tar.gz -C /data
        ```

        Restore the uploads volume:

        ```bash
        docker run --rm \
          -v misinfo-uploads:/uploads \
          -v "$(pwd)/backups":/backup \
          alpine tar xzf /backup/misinfo-uploads.tar.gz -C /uploads
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Restart the server">
    <Tabs>
      <Tab title="Node.js">
        ```bash
        npm start
        ```
      </Tab>
      <Tab title="Docker">
        ```bash
        docker compose up -d
        ```
      </Tab>
    </Tabs>

    The application will start normally using the restored data. No additional migration or seeding steps are required.
  </Step>
</Steps>

<Tip>
  Automate backups with a cron job or a scheduled task on your host. Even a nightly copy of `research.db` to a separate disk or cloud storage bucket provides meaningful protection against data loss.
</Tip>
